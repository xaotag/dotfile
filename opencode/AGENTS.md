# OpenCode 全局指令（精简版）

## 目标
- 默认使用中文回复，表达简洁、直接、可执行。
- 默认先做再说，不反复询问；只有高风险操作才先确认。
- 修改代码后说明关键变更点与原因，避免冗长解释。

## 工具与搜索策略
- 工具选择优先级：优先使用MCP，其次使用Skill，最后自行选择合适的工具
- 文件名搜索优先：`glob`；内容搜索优先：内置 `grep` 工具（底层为 ripgrep）。
- 需要 shell 搜索时优先：`fd` 和 `rg`。
- 禁止使用：`find`、`grep`（shell 命令）。若被拒绝，自动改用 `fd`/`rg` 或内置 `glob`/`grep` 工具重试。
- 文件读取优先使用 `read`，大文件用 `offset/limit` 分段读取。
- 非必要不启动子代理；能用基础工具解决就不要升级复杂度。

## 代码风格
- 使用清晰、可读的命名（驼峰或蛇形均可，遵循项目既有风格）。
- 拒绝过度工程化：优先直观线性实现；逻辑复用不足 3 次不强拆。
- 优先现代语法与类型能力（如 ES2022+、Python 3.12+）。
- 考虑边界条件；仅在必要处补充简短注释或类型说明。
- 前端样式默认使用 Tailwind CSS；无明确要求不引入额外样式方案。

## 操作确认规则

### 必须先确认（高风险）
- Git 破坏性命令：`git push --force`、`git reset --hard`、`git clean -fd`、危险 `rebase`。
- 高风险删除：`rm -rf`、删除配置/数据库文件、批量删除（5+ 文件）。
- 数据库破坏操作：`DROP`、`TRUNCATE`、大批量 `DELETE`、Schema 迁移。
- 关键配置变更：`package.json` 依赖变更、`tsconfig.json`、`.env*`、`docker-compose.yml`、CI/CD 配置。
- 新增或升级第三方依赖（尤其 major 版本）。

### 建议先提醒（中风险）
- 影响 5+ 文件的重构。
- API 签名或数据结构的 breaking change。
- 权限、安全、鉴权相关逻辑调整。

### 可直接执行（低风险）
- 小范围代码修复、文档更新、日志调整、测试补充、格式化。

```
- [后果2]

建议: [替代方案或降风险建议]

是否继续？请明确回复 'yes' 或 'no'
```
